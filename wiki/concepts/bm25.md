# BM25

## Definition

Okapi BM25 is a TF-IDF-based ranking function for document retrieval. Most widely used sparse retrieval method.

## Formula

BM25 score for document D given query Q:

```
score(D, Q) = Σ IDF(qi) * (f(qi, D) * (k1 + 1)) / (f(qi, D) + k1 * (1 - b + b * |D| / avgdl))
```

Where:
- `qi`: Query term i
- `f(qi, D)`: Term frequency in document
- `|D|`: Document length
- `avgdl`: Average document length
- `k1`: Term frequency saturation parameter (typically 1.2-2.0)
- `b`: Length normalization parameter (typically 0.75)
- `IDF(qi)`: Inverse document frequency

## IDF Calculation

```
IDF(qi) = log((N - n(qi) + 0.5) / (n(qi) + 0.5) + 1)
```

Where:
- `N`: Total documents
- `n(qi)`: Documents containing term qi

## Key Properties

### Term Frequency Saturation

Unlike raw TF-IDF, BM25 saturates term frequency:

- More occurrences increase score
- But effect diminishes (concave function)
- Prevents term spamming

### Length Normalization

Adjusts for document length:

- Long documents penalized (parameter `b`)
- Short documents favored
- Prevents bias toward long docs

### Parameters

**k1 (Term frequency saturation)**:
- Typical: 1.2-2.0
- Higher: More saturation (less TF impact)
- Lower: Linear TF (more TF impact)

**b (Length normalization)**:
- Typical: 0.75
- Higher: Strong normalization
- Lower: Weak normalization

## Implementation

```python
from rank_bm25 import BM25Okapi

# Tokenize corpus (lowercase for case-insensitivity)
tokenized_corpus = [doc.lower().split() for doc in documents]

# Build BM25 index
bm25 = BM25Okapi(tokenized_corpus)

# Score query
tokenized_query = query.lower().split()
scores = bm25.get_scores(tokenized_query)

# Get top-k documents
top_docs = bm25.get_top_n(tokenized_query, documents, n=10)
```

## Advantages

- Fast computation
- Handles exact keyword matching
- Proven effectiveness (decades of use)
- No training required
- Works on raw text (no embeddings)

## Disadvantages

- No semantic matching
- Vocabulary mismatch problem
- Case-sensitive (must normalize)
- Rare terms may have issues
- No context understanding

## Variants

- **BM25Okapi**: Standard implementation
- **BM25Plus**: Modified for term frequency saturation
- **BM25L**: Alternative length normalization

## Use Cases

- Keyword search
- Short queries
- Known terminology
- Quick initial retrieval
- Combined with dense methods ([[hybrid-retrieval|Hybrid Retrieval]])

## Integration with RAG

Often used as Stage 1 in [[multi-stage-retrieval|Multi-Stage Retrieval]]:

1. BM25 retrieves 20-50 keyword matches
2. Dense retrieval provides semantic matches
3. Combined in [[hybrid-retrieval|Hybrid Retrieval]]
4. Cross-encoder re-ranks final candidates

## Comparison with Dense

| Aspect | BM25 | Dense |
|--------|------|-------|
| Matching | Keywords | Semantic |
| Speed | Fast | Medium |
| Training | None | Required |
| Vocab mismatch | Problem | Handled |
| Exact terms | Strong | Weak |

## Best Practices

1. **Tokenization**: Lowercase text, split on whitespace
2. **Corpus size**: Works best with moderate to large corpus
3. **Query type**: Best for short, keyword queries
4. **Combine**: Use with dense for hybrid retrieval
