# Hybrid Retrieval

## Definition

Retrieval approach combining dense vector search and sparse vector methods (like BM25) to improve document relevance scoring.

## Components

### Dense Vector Retrieval

- High-dimensional floating-point vectors
- Captures semantic similarity
- Generated by embedding models (e.g., sentence transformers)
- Uses distance metrics like L2 or cosine similarity

### Sparse Vector Retrieval

- High-dimensional integer vectors (many zeros)
- Captures exact keywords and terms
- Often uses TF-IDF-based scoring
- BM25 is most common implementation

## Why Hybrid?

Neither approach is universally superior:

- **Dense excels at**: Semantic similarity, concept matching, paraphrases
- **Sparse excels at**: Exact keywords, specific terminology, rare terms

Combining both captures both semantic and lexical matches.

## Implementation

### BM25 Component

```python
from rank_bm25 import BM25Okapi

# Tokenize documents to lowercase
tokenized_corpus = [doc.lower().split() for doc in documents]
bm25 = BM25Okapi(tokenized_corpus)

# Get scores for query
tokenized_query = query.lower().split()
bm25_scores = bm25.get_scores(tokenized_query)
```

### Dense Component

```python
# Generate embeddings with mean pooling
query_embedding = generate_embedding(query)

# Use FAISS for efficient search
index = faiss.IndexFlatL2(dimension)
distances, indices = index.search(query_embedding, len(docs))

# Convert distance to similarity
dense_scores = 1 / (eps + distances)
```

### Score Combination

```python
# Normalize both to [0, 1]
bm25_scores = bm25_scores / max(bm25_scores)
dense_scores = dense_scores / max(dense_scores)

# Weighted combination
combined = alpha * dense_scores + (1-alpha) * bm25_scores
```

## Alpha Parameter

Controls balance between methods:

- `alpha = 1.0`: Pure dense retrieval
- `alpha = 0.5`: Equal balance
- `alpha = 0.0`: Pure sparse retrieval

Tune based on query characteristics and domain.

## Advantages

- Better recall than single method
- Handles both semantic and keyword queries
- More robust to query variations
- Complementary strengths cover weaknesses

## Disadvantages

- Computational overhead from two methods
- Requires normalization for fair combination
- Optimal alpha varies by query
- Needs both indices maintained

## Use Cases

- General-purpose RAG systems
- Mixed query types (semantic + keyword)
- Domain-specific terminology present
- High recall requirements
